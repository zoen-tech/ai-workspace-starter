# 02. 非エンジニアのためのGit / GitHub入門

> **要旨：** Gitの操作はAIに任せられるので暗記は不要。ただし「何が起きているか」を知っておくと、事故に気づける。

---

## このワークスペースでのGitの立ち位置

先に結論から言います。**日常のGit操作は、Claude Code / Codex に日本語で頼めば代行してくれます。**「最新にして」「コミットしてpushして」と言えば済みます。コマンドを暗記する必要はありません。

ではなぜ学ぶのか。**AIが何をしているか分かると、事故に気づけるから**です。たとえば「pushしたはずが他の人の変更を消していた」「共有すべきでないファイルを上げてしまった」——こうした事態は、仕組みを少し知っているだけで未然に防げます。この章は「暗記のため」ではなく「監督できるようになるため」に読んでください。

---

## そもそもGit / GitHubとは

### Git（ギット）

**ファイルの変更履歴を管理するツール。**

普段の仕事では、ファイルを編集したら上書き保存します。Gitを使うと、それに加えて **いつ・誰が・何を変えたか** がすべて記録され、いつでも過去の状態に戻せます。

> イメージ：「Googleドキュメントの変更履歴」のファイル版。間違えても過去のバージョンに戻せる。

### GitHub（ギットハブ）

**Gitで管理しているファイルを、インターネット上に保存・共有する場所。**

> イメージ：「Google Drive」のGit版。チームメンバーと同じファイルを共有できる。

Gitは自分のPCの中の仕組み、GitHubはそれをネット上に置いてチームと共有する場所、という関係です。（チームで使わず、自分のPCだけで完結させることもできます。その場合GitHubは不要です。）

### なぜGoogle DriveではなくGitHubか

- AIエージェント（Claude Code / Codex）がフォルダとファイルの構造を正確に理解できる
- 誰が何を変えたかの履歴が正確に残る
- メンバーごとにアクセス権を細かく設定できる（このリポジトリは全員／このリポジトリは担当者だけ、といった分け方）

## 基本用語

| 用語 | 意味 | たとえるなら |
|------|------|-------------|
| **リポジトリ（repo）** | ファイルの入れ物。1プロジェクト = 1リポジトリ | 共有フォルダ |
| **クローン（clone）** | GitHubから自分のPCにリポジトリをコピーすること | 共有フォルダをダウンロード |
| **コミット（commit）** | 変更を1つの区切りとして記録すること | ゲームのセーブ |
| **プッシュ（push）** | 自分のPCの変更をGitHubにアップロードすること | 編集を共有フォルダに戻す |
| **プル（pull）** | GitHubの最新の変更を自分のPCに取り込むこと | 共有フォルダの最新版を取得 |
| **ブランチ（branch）** | 本番に影響を与えず作業するための「別ルート」 | 下書きページで編集 |

## 日常の作業フロー

覚えるのはこの5ステップの流れだけで十分です（各操作はAIが代行します）。

```
1. 作業開始 → pull（最新を取得）
2. ファイルを編集
3. add（変更を記録の対象にする）
4. commit（変更を記録する）
5. push（GitHubにアップロードする）
```

ポイントは **最初に必ず pull すること**。他の人の変更を取り込んでから作業を始めないと、あとで衝突（コンフリクト）が起きやすくなります。「編集を頼む前に、まず最新にして」とAIに言う癖をつけると安全です。

## よく使うコマンド

AIに任せる場合でも、出てくるコマンドの意味が分かると安心です。**すべてターミナルで実行するコマンドです。**

### 最新の状態を取得する（作業前に必ず）

```bash
git pull
```

### 自分が何を変更したか確認する

```bash
git status
```

出力例：
```
modified:   report.md      ← 変更したファイル
```

### 変更を記録する（コミット）

```bash
git add .                       # すべての変更を記録の対象にする
git commit -m "月次レポートを更新"   # メッセージを付けて記録
```

> **コミットメッセージ**は「何をしたか」を簡潔に。日本語でOK。あとで履歴を見た人（未来の自分を含む）が分かるように書きます。
> 良い例：`クライアント一覧に新規案件を追加` / `料金プランの表を修正`

### GitHubにアップロードする（プッシュ）

```bash
git push
```

これで他のメンバーもあなたの変更を見られるようになります。

### 早見表

| やりたいこと | コマンド |
|-------------|---------|
| 最新を取得 | `git pull` |
| 変更を確認 | `git status` |
| 変更を記録 | `git add .` → `git commit -m "メッセージ"` |
| アップロード | `git push` |
| まとめて実行 | `git add . && git commit -m "メッセージ" && git push` |

## AIエージェントに任せる場合

Claude Code / Codex を使っていれば、チャットで頼むだけで上記を代行します。

- 「最新の状態にして」→ `git pull` してくれる
- 「変更をコミットしてpushして」→ `add` + `commit` + `push` してくれる
- 「report.md に今月の数値を追記して」→ 編集からコミット・pushまで一気にやってくれる

> つまり **コマンドを覚えなくても作業はできます。** ただしAIが「何を」「どのリポジトリに」pushしようとしているかは、実行前のAIの説明で軽く確認する習慣をつけましょう。それが唯一の「監督」です。

## 困ったときの対処

### 変更を取り消したい（まだコミットしていない場合）

```bash
git restore ファイル名     # 特定のファイルを元に戻す
git restore .             # すべての変更を元に戻す
```

### 「pushできない」と言われた

たいていは、他のメンバーの変更がGitHub上にある（＝手元が古い）のが原因です。

```bash
git pull    # 最新を取り込んでから
git push    # もう一度push
```

### よく分からないエラーが出た

**AIエージェントに「このエラーを解決して」と、エラー文をそのまま貼って頼む**のが一番早いです。それでも解決しなければ、チームの管理者にエラーのスクリーンショットを添えて相談してください。

## GitHubのWeb画面

コマンドを使わなくても、GitHubのWebサイト上でファイルの中身や変更履歴を確認できます。日常の編集はローカルのAIエージェントで行い、**Web画面は「確認用」**と位置づけると迷いません。

- ファイルを見る … リポジトリを開いてファイルをクリック
- 履歴を見る … ファイルを開いて **History** をクリック（いつ・誰が・何を変えたかが一覧で出る）
