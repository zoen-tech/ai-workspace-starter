# 04. ナレッジ蓄積の仕組み

> **要旨：** 日々のフィードバックを「原理原則」に昇華してナレッジ専用リポジトリに貯める。個別ルールを溜め込むのではなく、横展開できる原則に変えるのがコツ。

---

## なぜナレッジ専用リポジトリを分けるのか

業務の中でAIに出すフィードバック——「こう書いて」「これは分かりにくい」「この観点も見て」——には、**業務のノウハウが詰まっています**。しかしその多くは会話の中で発生し、そのまま流れて消えます。次のセッションにも、他のメンバーの作業にも活かされません。

これを防ぐために、**案件をまたいで使える「原理原則」だけを集める専用リポジトリ**を1つ用意します（例：`repo_company-knowledge`）。

なぜ専用リポジトリなのか。

- **案件横断で使えるから** … 特定のクライアントのリポジトリにノウハウを書くと、他の案件で参照できません。汎用的な原則は共通の置き場に集めるべきです
- **どこで作業していても書き込める／読み込める** … ワークスペース直下のコンテキストファイル（03参照）に「ナレッジは常にこのリポジトリに書く」と定めておけば、どのリポジトリで作業中でも一貫してここに貯まります
- **前章のコンテキストファイルとは役割が違う** … コンテキストファイルは「このリポジトリでの振る舞い方（ルール・手順）」、ナレッジリポジトリは「業務そのものの原理原則（考え方・方法論）」。前者は運用ルール、後者は知見、と分けて考えます

### AIのメモリ機能との違い

「それ、AIのメモリ機能でよくない？」という疑問はもっともです。Claude Code / Codex には会話から学んだことを覚えるメモリの仕組みがあり、**ひとりで・1つのツールだけで使っているうちは、実際メモリでかなりの部分が賄えます**。

それでもナレッジリポジトリを分ける理由は、メモリが「**その人の・そのツールの**」記憶でしかないからです。

| | AIのメモリ | ナレッジリポジトリ |
|---|---|---|
| 誰に効くか | 自分だけ | チーム全員のAIに効く |
| どのツールで効くか | そのツール専用 | Claude CodeでもCodexでも読める |
| 中身の確認・手直し | しづらい（AI任せ） | ただのファイルなので、レビュー・編集・履歴管理ができる |

ひとことで言えば、ナレッジリポジトリは**チームで共有できるAIの記憶**です。ひとり運用の間はメモリ中心で始めても構いません。チームで共有を始めるタイミングが、ナレッジリポジトリの価値が立ち上がるタイミングです。

## フィードバック → 原理原則 → 追記のループ

ナレッジは、次のループで自動的に貯まっていきます。

```
1. 通常の業務（どのリポジトリでも）
   ↓
2. ユーザーがフィードバックを出す
   例：「こう書いて」「分かりにくい」「〜も考えて」「〜ではなく〜」
   ↓
3. AIが検出 → 背後の「原理原則」を抽出 → ナレッジリポジトリの該当ファイルに追記
   （その場で追記する。毎回の確認は不要）
   ↓
4. コミット・push時にナレッジの変更も一緒に共有される
   ↓
5. 次のセッションでAIがナレッジを読み、過去の原則を踏まえて作業する
```

ポイントは **3の「原理原則を抽出」** です。フィードバックをそのまま書き写すのではなく、「なぜそう直したのか」を一段抽象化してから記録します。

## 個別ルールと原理原則の違い

ここがナレッジ蓄積の肝です。**そのとき限りの個別ルールを溜め込むと、ファイルが場当たり的な指示の山になり、他の場面で使えません。** 横展開できる原則に昇華してこそ、蓄積する価値が出ます。

### 悪い例：場面限定のルールをそのまま書く

> 「事業Aの案内文の冒頭には結論を先に書く」

これは事業Aの案内文にしか使えません。別の業務では参照されず、似た指摘が何度も繰り返されます。

### 良い例：横展開できる原則に昇華する

> 「読み手が忙しい前提の文章は、結論を先頭に置く。理由・詳細はその後に回す。
>  背景例：事業Aの案内文で『結論を先に』と指摘された」

こう書くと、案内文でも報告でもメールでも——**忙しい読み手に向けたあらゆる文章**に適用できます。元になった具体的な指摘は「背景例」として残し、原則の出自が分かるようにします。

### 抽象化のやり方（3ステップ）

1. **「なぜそうすべきか？」を問う** … 個別の指示の背後にある理由を言葉にする
2. **業種・場面の限定を外す** … 「事業Aの案内文」→「忙しい読み手向けの文章」のように、より広い言い方に置き換える
3. **具体例は背景例として残す** … 原則がどこから来たかを補足として添える

もし背後の原則がうまく言語化できないときは、無理に一般化せず、AIから「こういう原則ですか？」と仮説を確認してもらい、正しい原則を貯めます。

## エントリの書式

ナレッジは、領域ごとのファイル（例：営業／ライティング／分析、といった自分の業務に合った分類）に分けて貯めます。1つのエントリは次の形が扱いやすいです。

```markdown
### 原則のタイトル（簡潔に）
- **原則**: 横展開できる形で言い切る（何をすべきか／すべきでないか）
- **背景例**: この原則が生まれた具体的な場面（顧客固有情報は書かない）
- **日付**: YYYY-MM-DD
```

### 蓄積のルール

- **顧客固有の情報（社名・URL・個人名など）は書かない。** ナレッジは汎用的な知見として残す。固有情報が混ざると共有できなくなります
- 新しいエントリは、該当する領域ファイルの末尾に追記する
- 既存エントリと本質的に同じ原則なら、新規に作らず**背景例を1行足すだけ**でよい（同じ原則が複数の場面で確認された、という強化になります）
- 1つのファイルが長くなりすぎたら、見出し単位で別ファイルに分割する
- 追記は基本その場で行い、毎回の確認は取らない。不要なエントリは後から履歴を見て削除できるので、「貯め漏らす」より「多めに貯める」方を優先します
