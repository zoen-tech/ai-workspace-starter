# 03. コンテキストファイル（CLAUDE.md / AGENTS.md）の設計

> **要旨：** コンテキストファイルは、AIが起動時に自動で読む「常設の指示書」。階層で置き、人の回答をもとに書き、使いながら育てる。

---

## コンテキストファイルとは

**コンテキストファイルとは、AIエージェントが起動時に自動で読み込む指示書**です。毎回チャットで説明しなくても、AIがそこに書かれたルール・構成・手順を前提に動いてくれます。

- Claude Code → `CLAUDE.md`
- Codex → `AGENTS.md`

中身はどちらも同じ考え方の指示書で、ファイル名が違うだけです。**両方のツールを使う人は、同じ内容の `CLAUDE.md` と `AGENTS.md` を両方置きます**（この点は後述します）。

チャットでの指示が「その場かぎりの口頭指示」だとすれば、コンテキストファイルは「壁に貼ってある運用マニュアル」です。AIは作業のたびにこれを読み直すので、一度書けばセッションをまたいで効き続けます。

---

## 階層構造：2つの層に分ける

コンテキストファイルは1つではありません。**役割の違う2つの層**に分けて置きます。

```
your-name-workspace/
├── CLAUDE.md / AGENTS.md     ← ① ワークスペース直下＝「振り分け役」
├── repo_business-a/
│   └── CLAUDE.md / AGENTS.md ← ② リポジトリ直下＝「個別ルール」
├── repo_business-b/
│   └── CLAUDE.md / AGENTS.md ← ② リポジトリ直下＝「個別ルール」
└── repo_knowledge/
    └── CLAUDE.md / AGENTS.md ← ②
```

### ① ワークスペース直下 ＝ 振り分け役

全体の地図と、共通ルールを書きます。役割は主に「**どの依頼が来たら、どのリポジトリで作業すべきか**」をAIに判断させることです。

- どんなリポジトリがあり、それぞれ何のためのものか（一覧表）
- 作業開始時の手順（依頼内容 → 対象リポジトリを特定 → そのリポジトリのコンテキストファイルを読む）
- どのリポジトリで作業していても守る共通ルール（応答言語、機密情報をコミットしない、報告フォーマットなど）

### ② リポジトリ直下 ＝ 個別ルール

そのリポジトリ**固有**の事情だけを書きます。ここに全体ルールを重複させると、変更のたびに全リポジトリを直す羽目になるので、共通ルールは①に集約し、②には個別のことだけを書くのが鉄則です。

具体的には、次の「4項目」を書きます。

---

## 新規プロジェクトの4項目（Why / Structure / Rules / Workflow）

新しいリポジトリ（プロジェクト）を作るとき、そのリポジトリ直下のコンテキストファイルには次の4項目を書きます。

| 項目 | 何を書くか |
|------|-----------|
| ① **Why**（目的） | このリポジトリの存在理由。何のための場所で、何を置くか |
| ② **Structure**（構成） | フォルダ構造と各ディレクトリの役割 |
| ③ **Rules**（ルール） | AIに守らせたいルール・禁止事項（コミットしてはいけない情報など） |
| ④ **Workflow**（手順） | よくある作業の手順・パイプライン |

### なぜ「人にヒアリングして書く」のか

**AIに自動生成させず、人の回答をもとに書く**——これがこのワークスペースの重要な原則です。

AIが自動生成したコンテキストファイルは、フォルダ構成から機械的に推測した「それらしい説明」になりがちで、**そのプロジェクトの本当の目的・背景・落とし穴（＝人の頭の中にしかない情報）が抜け落ちます**。抜けたまま置かれた指示書は、AIをかえって間違った方向に導き、「何も書いていない状態より精度が下がる」ことすらあります。

だから4項目は、AIがあなたに質問し、**あなたの回答をもとに**書きます。AIの想像で埋めさせないこと。答えが決まっていない項目は、無理に埋めず「（未記入。後日追記する）」と見出しだけ残しておけば十分です。

> AIに新規リポジトリの作成を頼むと、AI側からこの4項目を質問してきます。答えられる範囲で答えれば、AIがそれを整形して書き込みます。

---

## コンテキストファイルの育て方

コンテキストファイルは、最初に完璧に書くものではありません。**使いながら育てるもの**です。

### フィードバックのたびに追記する

AIの作業結果に対して「そうじゃなくて、こうして」「この形式で出して」と直したくなったら、それは**次からAIに守ってほしいルール**である可能性が高いです。その場のやり直しで終わらせず、該当するコンテキストファイルに一文追記しておくと、同じ指摘を繰り返さずに済みます。

- そのリポジトリ固有のこだわり → そのリポジトリの `Rules` / `Workflow` に追記
- どのリポジトリでも守ってほしい共通のこと → ワークスペース直下の共通ルールに追記

> 「業務のノウハウ・原理原則」として貯めるべきフィードバックは、コンテキストファイルではなく専用のナレッジリポジトリに貯めます。その仕分けは次章（04）で扱います。

### ルールが増えたら整理する

追記を続けると、いずれ箇条書きが増えて読みにくくなります。そうなったら、似たルールをまとめる・見出しで区切る・順序を整える、といった整理をします。整理も「このコンテキストファイルを読みやすく整理して」とAIに頼めます。

**書きすぎにも注意**します。矛盾したルールや、細かすぎて滅多に使わないルールが増えると、AIがどれを優先すべきか迷います。「常に守ってほしい重要なこと」に絞り、例外的な指示はその場でチャットする、というバランスが実用的です。

---

## Claude Code / Codex 両対応の注意

両方のツールを使う場合、**`CLAUDE.md` と `AGENTS.md` は同一内容に保ちます**。

- 片方だけ更新すると、使うツールによってAIの振る舞いが変わり、原因の分かりにくい不整合になります
- どちらか一方を編集したら、必ずもう片方にも同じ変更を反映します
- 運用を簡単にするため、コンテキストファイル自身に「**片方を更新したら必ずもう片方も更新する**」という一文を書いておくと、AI自身がそれを守ってくれます

片方のツールしか使わないなら、そのツールのファイル（`CLAUDE.md` か `AGENTS.md`）だけで構いません。
