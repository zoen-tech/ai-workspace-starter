# 01. AIワークスペースの設計思想

> **要旨：** なぜ「1つの親フォルダに全業務を集める」のか。その1点を理解すれば、以降のすべてのルールの理由が分かる。

---

## AIワークスペースとは何か

AIワークスペースとは、**あなたの業務データをすべて1つの親フォルダに集約し、AIエージェント（Claude Code / Codex）に全体を見せながら日常業務を行う作業空間**のことです。

普通のフォルダと違うのは、置き場所ではなく「AIと一緒に働くために設計されている」点です。施策の立案、資料づくり、情報の整理、原稿の作成——こうした日々の仕事を、AIに文脈ごと渡して進めるための基盤になります。

> **用語：AIエージェント** … 人の指示を受けて、ファイルの読み書きやコマンドの実行までを自分でこなすAI。本書では代表として Claude Code（`claude` で起動）と Codex（`codex` で起動）で説明しますが、Cursor・Antigravity・VS Code（GitHub Copilot）など、フォルダを開いて作業できるAIエージェントなら同じ考え方で使えます。いずれも、開いたフォルダの中身を読んで作業します。

---

## なぜ「1つの親フォルダに集約」するのか

理由はシンプルで、**AIは「見えている範囲」でしか判断できない**からです。

業務データがあちこちのフォルダやクラウドに散らばっていると、AIはそのつど「これは何のファイル？」「この案件の背景は？」と聞き直すことになります。逆に、関連する業務が1つの親フォルダにまとまっていれば、AIはフォルダ構成そのものから「どんな事業があり、どんな案件が動いていて、どんなルールで運用されているか」を読み取れます。

つまり親フォルダは、**AIにとっての「全体地図」**です。地図が1枚にまとまっているほど、AIは的確に動けます。

---

## 開発環境ではなく「業務基盤」である

「フォルダをGitで管理する」と聞くと、エンジニアの開発環境を思い浮かべるかもしれません。しかしこのワークスペースは**コードを書くための環境ではなく、業務そのものを回すための基盤**です。

- 情報・データ・ナレッジを**蓄積する場所**であり、
- 同時に、その情報を使って日々の業務を**実行する場所**でもある

Gitを使うのは「エンジニアだから」ではなく、**変更履歴が残る／AIが構造を理解しやすい／チームで権限を分けて共有できる**という3つの実利のためです（Gitそのものは次章で解説します）。

> **用語：Git** … ファイルの変更履歴を記録するしくみ。「いつ・誰が・何を変えたか」がすべて残り、過去の状態に戻せる。Googleドキュメントの変更履歴のファイル版だと考えてください。

## 5つの設計原則

このワークスペースは、次の5つの原則で構成されています。

### 1. リポジトリ = 業務領域の境界

**1つのリポジトリに全部を詰め込まず、業務領域ごとに独立したリポジトリに分ける。**

> **用語：リポジトリ（repo）** … Gitで管理する1つのフォルダのこと。1プロジェクト = 1リポジトリが基本。

分ける目的は2つあります。

- **機密性の分離** … クライアントや財務など、見せる相手を限定したい情報を、独立したリポジトリにして権限を個別に設定できる
- **文脈の分離** … AIが「いま自分はどの業務の文脈にいるか」を取り違えないようにする。事業Aの作業中に事業Bの事情が混ざらない

### 2. ローカルまとめフォルダ

Git管理はしないが、関連リポジトリをグルーピングしたい——そういう用途のために、**プレフィックスなしのローカル専用フォルダ**を使います（例：ダウンロードの一時置き場、複数サイトのまとめ場所）。まとめフォルダ自体はGit管理せず、中に入れる個々のリポジトリごとに権限を分けます。

### 3. `repo_` プレフィックス規則

**Git管理するフォルダには `repo_` を付け、ローカル専用フォルダには付けない。**

```
your-name-workspace/
├── repo_business-a/     ← Gitリポジトリ（履歴が残る・共有できる）
├── repo_knowledge/      ← Gitリポジトリ
├── downloads/           ← ローカル専用（Git管理しない）
└── gdrive/              ← ローカル専用
```

名前を見ただけで「これは共有される／されない」が一目で分かります。AIも人も、うっかりローカル専用フォルダをコミットしたり、逆に共有すべきものを手元に留めたりする事故を防げます。

### 4. 再現可能な環境

セットアップの手順・設計情報を1箇所（`workspace-setup` リポジトリ）に集約しておくと、**新しいメンバーが同じ構造のワークスペースを再現できます**。手順が人の頭の中にしかない状態を避け、「clone してスクリプトを実行すれば同じ環境になる」状態を保ちます（チーム運用の詳細は 05 で扱います）。

### 5. 同一階層には同一粒度

**1つのフォルダの中には、できるだけ同じ粒度（抽象度）のものだけを置く。**

たとえば「個別の議事録ログ」と「それらを要約したサマリー」が同じ階層に混在すると、見通しが悪くなります。粒度が違うものはサブフォルダで分けます。

```
# 見通しが悪い例
meetings/
├── 2025-10-16.md    ← 個別ログ（何十件も並ぶ）
├── 2025-10-24.md
├── timeline.md      ← サマリー（ログに埋もれる）
└── key-decisions.md ← サマリー

# 粒度で分けた例
meetings/
├── logs/            ← 個別ログはここにまとめる
│   ├── 2025-10-16.md
│   └── ...
├── timeline.md      ← サマリー系は同じ階層に並ぶ
└── key-decisions.md
```

絶対ルールではありません。ファイルが増えて見通しが悪くなってきたら分ける、という判断の指針です。

---

## リポジトリはどう分けるか

設計で最も悩むのが「業務をどこで区切るか」です。基準は原則1の通り、**扱う情報の機密度・関わる人・業務の文脈が違うものは分ける**です。

判断に迷ったときの目安：

- **細かすぎのサイン** … 1つのリポジトリに数ファイルしか入らない見込み → 近い領域と統合する
- **粗すぎのサイン** … 1つのリポジトリの中に、見せたい相手が違う情報や、まったく別の事業の情報が同居している → 分割する

例（抽象名で示します）：

| 分け方 | リポジトリ例 | 理由 |
|--------|------------|------|
| 事業ごと | `repo_business-a` / `repo_business-b` | 文脈がまったく異なる |
| クライアントごと | `repo_org-client-x` / `repo_org-client-y` | 機密性が高く、担当者だけに見せたい |
| 機密度で分離 | `repo_org-finance` | 財務・人事など、見られる人を強く限定したい |
| 全員共通 | `repo_org-common` | テンプレートや共通情報など、全員が使う |

最初から完璧に分ける必要はありません。**まず大きく分けて始め、運用しながら「混ざって不便になったら分ける／細かすぎたら統合する」**で十分です。分割・統合は後からでもできます。
