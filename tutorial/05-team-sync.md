# 05. チームでの運用と同期

> **要旨：** 「置くべきリポジトリの一覧」を1枚のマニフェストで管理し、同期スクリプトに任せる。メンバーは自分に見える範囲だけのワークスペースが自動で組み上がる。

> **Note:** この章はチームで使う場合の話です。自分1人・GitHubなしで使うなら、読み飛ばして構いません（人が増えたときに戻ってくればOK）。

---

## マニフェスト（repos.tsv）＝ 単一の正

チームでワークスペースを共有すると、すぐに「どのリポジトリを、どのフォルダ名で置けばいいのか」が分からなくなります。ローカルのフォルダ名は、GitHub上のリポジトリ名から機械的には決まりません（`client-x` を `repo_org-client-x` に、`common` を `repo_org-common` に、といった変換は運用ルール次第で不規則です）。

そこで、**「置くべきリポジトリの一覧」を1枚のファイルに明示的に書き、それを唯一の正しい情報源（＝単一の正）とします**。これがマニフェスト `repos.tsv` です。`workspace-setup` リポジトリの中に置きます。

中身は1行1リポジトリの単純な表です（タブ区切り）。

```
# 形式: <org/repo><TAB><ローカルパス><TAB><区分>
your-org/your-org-common     repo_your-org-common     common
your-org/your-org-knowledge  repo_your-org-knowledge  common
your-org/workspace-setup     repo_workspace-setup     common
your-org/business-a          repo_business-a          business
your-org/client-example      repo_your-org-client-example  business
your-org/finance             repo_your-org-finance    restricted
```

- **org/repo** … GitHub上のリポジトリ
- **ローカルパス** … 各メンバーのPC上に置くフォルダ名（`repo_` プレフィックス付き）
- **区分** … アクセス範囲の分類（例：`common`＝全員／`business`＝案件単位で権限付与／`restricted`＝限定／`personal`＝各自のものに置き換え）。区分は自由に定義できます

このファイルさえ正しければ、フォルダ名がバラつく事故は起きません。

## sync-repos.sh の動き

マニフェストを見て、各メンバーのPCにリポジトリを揃えるのが同期スクリプト `sync-repos.sh` です。**ターミナルで実行**します。挙動はシンプルです。

- **clone済みのリポジトリはスキップ** … すでに手元にあるものには触れない（上書きしない）
- **権限のないリポジトリはスキップ** … そのメンバーにアクセス権がないもの（例：財務など限定リポジトリ）は「権限なし」で飛ばす

この2つの結果、**メンバーごとに「自分に見える範囲だけのワークスペース」が自動で組み上がります**。全員が同じマニフェストを使っているのに、財務リポジトリは担当者のPCにだけ現れ、他の人のPCには現れません。権限設定（GitHub側）とマニフェスト（一覧）を分業させているのがポイントです。

```
同じ repos.tsv を実行しても……
├─ 管理者のPC   → 全リポジトリが揃う
├─ 案件担当のPC → 担当案件＋共通リポジトリだけ揃う（財務はスキップ）
└─ 新メンバーPC → 共通リポジトリだけ揃う
```

## 新リポジトリ追加時の運用フロー

新しいリポジトリが増えたとき、全員のワークスペースに確実に行き渡らせる手順です。

### 管理者（リポジトリを作った人）

1. GitHub上にリポジトリを作り、関与メンバーにアクセス権を付与する
2. **マニフェスト `repos.tsv` に1行追記する**（org/repo・ローカルパス・区分）
3. `workspace-setup` リポジトリをコミット＆push
4. メンバーに「新リポを追加した。同期スクリプトを実行して」と通知する

### メンバー（通知を受けた人）

マニフェストを最新化してから同期スクリプトを実行するだけです。**ターミナルで実行：**

```bash
git -C repo_workspace-setup pull      # マニフェストを最新化
bash repo_workspace-setup/scripts/sync-repos.sh   # 未cloneのリポだけ取り込む
```

未cloneのリポジトリだけが正しいフォルダ名で追加され、既存のものには触れません。

## workspace-setup リポジトリを自チーム用に持つ意味

マニフェストや同期スクリプトを置く `workspace-setup` リポジトリは、**チームのワークスペースを再現可能にするための中枢**です。持つ意味は2つあります。

- **テンプレート原本の置き場** … ワークスペース直下のコンテキストファイル（`CLAUDE.md` / `AGENTS.md`）やリポジトリ用テンプレートの「原本」をここに置き、メンバーはここからコピーして使います。原本が1箇所にあることで、全員の初期設定が揃います
- **セットアップの再現性** … 「clone → 同期スクリプトを実行」という短い導線で、新メンバーが同じ構造のワークスペースを組み立てられます。手順が人の記憶に依存しなくなります

### ワークスペース直下のコンテキストファイルを原本に反映する

ここで1つ運用上の注意があります。**ワークスペース直下のコンテキストファイルは、各メンバーのローカル専用（Git管理しない）**です（メンバーごとに手元の事情が少し違うため）。

そのため、共通で効かせたいルールを誰かが自分の手元のコンテキストファイルに追記しても、それだけでは他のメンバーに伝わりません。**全員に効かせたい変更をしたら、`workspace-setup` リポジトリ内のテンプレート原本にも同じ内容を反映してコミット＆pushする**——このひと手間で、次にメンバーが原本を取り込んだときに全員へ行き渡ります。「手元で改善したら原本にも戻す」を習慣にしてください。

## 拡張パターン

ここまでが基本構造です。運用が回り始めると、次のような発展が考えられます（いずれも固有の仕組みに縛られない一般的な方向性です）。

定型業務——たとえば毎月の同じ形式のレポート作成や、決まった手順のデータ整形——は、手順をコンテキストファイルに書いておくだけでなく、**再利用可能な「スキル」や「コマンド」として切り出す**とさらに効率化できます。AIエージェントには、よく使う手順を名前付きの手続きとして登録し、ひと言で呼び出せるようにするしくみがあります。また、チャットや会議の記録を定期的に要約してリポジトリに蓄積する、といった**情報の自動整理**も、同じ発想の延長で組めます。まずは基本構造を回し、繰り返し発生する作業が見えてきたら、その部分だけを自動化・スキル化していくのが無理のない進め方です。
