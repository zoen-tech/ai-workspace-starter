### 応答・報告フォーマット

多忙な読み手が短時間で「何が起きて、自分は何をすればいいか」を判断できることを最優先に構成する。

**共通原則：どんな報告も冒頭2行で宣言する**

```
【タグ】結論を一文で
▶あなたのアクション：なし／確認のみ／1〜3から選択／〜の判断
```

**4類型**

| タグ | 場面 | 構成 |
|------|------|------|
| 【完了】 | 作業が終わった | 結論（何をした＋結果）→ 確認ポイント → 補足（読まなくてよい詳細） |
| 【相談】 | 判断・選択が必要 | 結論 → 背景（最大3点）→ 推奨（1つ＋理由1文）→ 選択肢 1/2/3 |
| 【報告】 | 問題・遅延が発生 | 結論（何が起きた）→ 現状/原因 → 対応案（推奨付き）→ 影響範囲 |
| 【共有】 | 情報のみ | 結論のみ3行以内。アクション不要と明示 |

**ルール**

1. 冒頭サマリは10行以内。判断に必要な材料はすべてサマリに入れ、詳細は「補足」以下に隔離する
2. 判断を求めるときは判断項目を質問文で書く（「Aで進めてよいか？」）
3. 選択肢には必ず推奨を1つ付ける。選択肢の記号は数字（1/2/3）を使う
4. 経過・試行錯誤の説明は求められない限り書かない

### ファイル/フォルダの移動・リネーム・削除時のチェックリスト

ファイル構造を変更する時は、必ず以下を確認してからコミットすること。旧パスを参照しているものが残ると、リンク切れ・自動化の停止などの障害が起きる。

1. 旧パスを grep で全文検索して参照箇所を洗い出す
2. 特に確認するもの：CI/自動化の設定（`.github/workflows/` 等）・README等のパス記載・Markdown間の相対リンク・スクリプト内のパス
3. コミット前に `git status` で意図しない変更がないか確認する
