# ナレッジベース

## 概要
日々の業務で得られたフィードバックや修正指示から、**業種・業務を問わず横展開できる原理原則**を抽出して蓄積するリポジトリです。
個別の「こうして」という指示をそのまま貯めるのではなく、その背後にある「なぜそうすべきか」を言語化して記録します。

## 設計思想
- **原則ファースト**: 個別ルールではなく、その背後にある原理原則を記録する
- **横展開可能**: 特定の業種・場面に限定されない抽象度で書く
- **根拠は残す**: どういう具体例からその原則が導かれたかを補足として残す

## ファイル構成
トピック別に Markdown ファイルを分けて管理します。ファイルの切り方は自分たちの業務領域に合わせて自由に決めてください。
1ファイルが長くなりすぎたら、見出し単位でファイルを分割します。

（例）
- `writing.md` — 文章表現・トーンのルール
- `analysis.md` — 分析の品質基準（深さ・視点・網羅性）
- `process.md` — 業務の進め方・段取り

## エントリのフォーマット

```markdown
### 原則のタイトル（抽象的・横展開可能な表現で）
- **ルール**: 業種・業務を問わず適用できる考え方。「なぜそうすべきか」の本質
- **背景**: この原則が確認された具体的な場面（顧客固有情報は含めない）
- **日付**: YYYY-MM-DD
```

### フォーマット補足
- **ルール**は「〜の場面では」のような限定を避け、広く適用できる表現にする
- **背景**は原則の出自を示す補足情報。「こういうケースで確認された」程度でよい

## 蓄積のルール
- **原理原則に昇華してから書く**: 「なぜそうすべきか」を問い、個別ルールを抽象化してから記録する
- **顧客固有の情報を含めない**: 社名・URL・担当者名など、特定の相手が識別できる情報は書かない
- **同じ原則なら背景例の追記のみ**: 既存エントリと本質的に同じ原則の場合は、新規エントリを作らず背景例を追記するだけでよい
- 新しいエントリは該当セクションの末尾に追記する
- ファイルを追加・分割したら、この README のファイル構成も更新する
