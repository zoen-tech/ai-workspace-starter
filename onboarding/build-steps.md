# 構築手順（AIエージェント向け）

ヒアリング完了・設計承認後、以下の順で構築する。**すべてワークスペースフォルダ側に作成する**（このリポジトリは変更しない）。

## Step 1：ワークスペースフォルダの作成

```bash
mkdir ~/[名前]-workspace
```

Q1-2 で別の場所を指定された場合は、そのパスに読み替える（以降のステップも同様）。

作成したら `setup-notes.md` をこの直下に書き出す（ヒアリング回答＋進捗）。

## Step 2：ワークスペース直下のコンテキストファイル

`templates/workspace-context.md` をベースに、`{{ }}` プレースホルダをヒアリング結果で置換して生成する。

- Claude Code → `CLAUDE.md`
- Codex → `AGENTS.md`
- 両方 → 同一内容で両方を生成し、「片方を更新したらもう片方も更新する」旨をファイル内に明記する（テンプレートに該当行あり）

テンプレートの各セクションは、ヒアリングで該当しなかったもの（チーム同期など）を削除してよい。**逆に、テンプレートにない独自ルールをユーザーが持っている場合は「基本ルール」に追記する。**

プレースホルダのうちヒアリングで直接聞いていないもの（機密情報の種類など）は、AIの想像で埋めず、置換前にユーザーに確認する。

生成後、未使用の `{{IF_...}}` ブロック・未置換のプレースホルダ・テンプレート冒頭のHTMLコメントが残っていないことを必ず確認する。

## Step 3：リポジトリ群の作成

分割案の各領域について、以下を行う。

1. フォルダ作成：`repo_[リポジトリ名]`（ワークスペース直下）
2. Git設定：
   - **ローカルのみ（Q3-1=1または2、標準の流れ）**：`git init` のみ。GitHub側には何も作らない。履歴はローカルに残るため、後からGitHubへ履歴ごと移行できる
   - **GitHub利用（Q3-1=3）**：`gh repo create [org または ユーザー名]/[リポジトリ名] --private` でリモートを作成してから clone またはリモート紐付けを行う。**リポジトリ作成は1件ずつユーザーに確認してから実行する。** 公開する明確な理由がない限り private とする
3. リポジトリ直下のコンテキストファイル：`templates/repo-context.md` の4項目（Why / Structure / Rules / Workflow）を**ユーザーにヒアリングして**生成する
   - 全リポジトリを一度にやると疲れるため、まず主要な1〜2個で型を見せ、残りは「後日、各リポジトリで最初に作業するときに作る」でよいと案内する
   - 未記入の項目は空欄のまま見出しだけ残す（後日追記できる状態にする）
4. 初回コミット（GitHub利用時は push まで）

## Step 4：workspace-setup リポジトリ（チーム利用の場合のみ）

チームメンバーが同じ構造のワークスペースを再現できるよう、ユーザー自身の workspace-setup リポジトリを作る。

1. `repo_workspace-setup` フォルダを作成し、GitHubにもリポジトリを作成する
2. `templates/repos.tsv.template` → `scripts/repos.tsv` として配置し、Step 3 で作った全リポジトリを記載する
3. `templates/scripts/sync-repos.sh` と `templates/scripts/setup-workspace.sh` を `scripts/` にコピーする
4. `templates/workspace-context.md` から生成したコンテキストファイルを、テンプレート原本として `templates/CLAUDE.md`（Codex利用時は `templates/AGENTS.md`）に格納する（メンバー配布用マスター。`setup-workspace.sh` がこのパスからコピーする）。**利用するAIのテンプレートだけを置くこと**（templates/ に置いたものがそのままメンバーに配布される。Q0-1で「両方」の場合は両方置く）
5. メンバー向け README を書く（clone → `setup-workspace.sh` 実行、の導線）
6. コミット＆push

【個人利用の場合は省略してよい。ただし「リポジトリが増えてきたら（目安5個以上）、または人を増やすときに導入するとよい」と案内する】

## Step 5：ナレッジリポジトリ

Q4-2 で決めた構成で `repo_[会社/チーム名]-knowledge` を作成する（Q4-2で不要とされた場合はこのStepをスキップ）。

1. `templates/knowledge/README.md` を配置（エントリの書式・蓄積ルールの説明）
2. 領域ごとの空ファイル（見出しのみ）を作成する
3. ワークスペースのコンテキストファイルにはナレッジ蓄積ルールのセクションが含まれている（Step 2 のテンプレートに同梱）。蓄積先ファイル一覧をヒアリング結果に合わせて書き換えること

## Step 6：ローカル専用フォルダ

Q4-3 で決めたフォルダ（プレフィックスなし）を作成する。Git管理しないため、ワークスペースのコンテキストファイルの構成ルール表に役割を明記しておく。

## Step 7：動作確認と引き渡し

`checklist.md` に従って確認し、結果をユーザーに報告する。最後に：

1. `tutorial/` の読み進め方を案内する（設計思想 → 日常の使い方の順）
2. 「このワークスペースで最初にやる業務」を1つ選んでもらい、実際にAIと一緒にやってみることを提案する（定着の最重要ステップ）
3. （ローカルのみ運用の場合）「今はバックアップがない状態。共有相手が現れたとき、またはバックアップが欲しくなったときが GitHub 導入のタイミングで、AIに頼めば履歴ごと移行できる」ことを最後に一度伝える
