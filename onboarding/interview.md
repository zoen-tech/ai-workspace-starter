## フェーズ1：ワークスペースの基本

**Q1-1.** 「ワークスペースのフォルダ名に使う名前を教えてください（`[名前]-workspace` という形になります。ローマ字の苗字を推奨）」

**Q1-2.** 「ワークスペースをどこに置きますか？」——以下のメリット・デメリットを提示した上で選んでもらう。
1. ホームフォルダ直下（推奨）
2. デスクトップ
3. 書類（Documents）
4. その他の場所

提示する比較（ユーザーの環境・理解度に合わせて言い換えてよい）：

| 場所 | メリット | デメリット |
|------|---------|-----------|
| ホームフォルダ直下（Macなら Finder の家アイコン＝自分の名前のフォルダ、`~/`） | デスクトップの整理・片付けに巻き込まれず、誤削除されにくい／iCloud等のクラウド同期の対象外で、Git管理と衝突しない／パスが短く安定 | Finderからは一階層たどる必要がある（フォルダをFinderサイドバーにドラッグしてピン留めすれば解決） |
| デスクトップ | 常に目に入るので存在を忘れない・すぐ開ける | デスクトップが散らかったとき、整理のつもりで誤って移動・削除しやすい／Macで「デスクトップと書類」のiCloud同期が有効だと、Git管理とクラウド同期が衝突して壊れやすい |
| 書類（Documents） | 書類フォルダから探す習慣の人には自然 | デスクトップと同じくiCloud同期の対象になっていることが多く、Git管理と衝突しやすい |

【判断材料として伝える：日常の操作はAIエージェント（ターミナル）経由なので、Finderでフォルダを探しに行く頻度は実際には低い。「見つけやすさ」より「壊れにくさ・巻き込まれにくさ」を優先するのが推奨。デスクトップ／書類を選ぶ場合は、iCloud同期（システム設定 → Apple ID → iCloud Drive →「デスクトップ」と「書類」）がオフであることを必ず確認する。OneDrive・Dropbox等の同期フォルダ配下も同じ理由で避ける】

## フェーズ2：業務領域の洗い出し（リポジトリ分割の設計）

このフェーズが設計の核心。時間をかけてよい。

**Q2-1.** 「あなた（またはあなたの会社）の業務を大きな塊で分けると、どんな領域がありますか？事業・部門・業務の種類など、思いつくまま挙げてください」

【回答を受けて深掘りする。目安：1つの領域に「扱う情報の機密度」「関わる人」「業務の文脈」の違うものが混ざっていれば分割を提案し、逆に細かすぎる場合（1領域に数ファイルしか入らない見込み）は統合を提案する】

**Q2-2.** 「クライアント（顧客）ごとに管理したい情報はありますか？」

【ある場合：`client-[英語名]` 命名規則を適用し、クライアントごとに独立リポジトリとする方針を説明する。クライアント情報は機密性が高く、担当者だけがアクセスできる形にすべきという理由も添える】

**Q2-3.** 「財務・人事・個人の記録など、特に機密度が高く、見られる人を限定したい領域はありますか？」

【ある場合：独立リポジトリに分離する。チーム利用ならアクセス権を個別設定できることを説明する】

【フェーズ2の締め：回答をもとにリポジトリ分割案（リポ名・ローカルフォルダ名・内容・アクセス範囲の表）を作成して提示し、合意を得る。原則は3つ — ①リポジトリ＝業務領域の境界 ②機密性の違うものは同居させない ③同一階層には同一粒度】

## フェーズ3：共有とバックアップ（Git / GitHub）

**Q3-1.** 「チームメンバーとの共有について、今の状況に近いものはどれですか？」
1. 今後チームで共有する予定だが、まずはひとりで始める（推奨）
2. 当面は自分ひとりで使う（共有の予定はまだない）
3. 最初からチームで共有したい（GitHubを使う）

【1・2の場合：各リポジトリは `git init` のみ（ローカルのGit管理だけ、リモートなし）で構築する。GitHubアカウントも認証も不要になり、セットアップが大幅に短くなる。ユーザーには次の2点を伝える：①変更履歴はローカルGitで最初から残るので、後からGitHubに載せるときは履歴ごとそのまま移行できる（AIに頼めばよい。`tutorial/05-team-sync.md` 参照）②GitHub導入のタイミングは「共有したい相手が現れたとき」または「バックアップが欲しくなったとき」（ローカルのみの間はPC故障でワークスペースごと失われるため）。マニフェスト（repos.tsv）と同期スクリプトも省略する。→ Q3-2以降はスキップしてフェーズ4へ】

【1の場合はさらに：将来の共有を見据え、フェーズ2のリポジトリ分割で「誰にどこまで見せるか」の境界を特に意識して設計する（後からの分割よりも最初から分けておく方が楽）。また、GitHub導入時にチームで使うなら個人アカウントの共有ではなく GitHub Organization の作成が推奨であることを、このタイミングで一言伝えておく】

**Q3-2.** （Q3-1が3の場合のみ）「GitHubの利用状況を教えてください」
1. 会社・チームの GitHub Organization がある
2. 個人アカウントのみ
3. アカウントがまだない

【3の場合：GitHubアカウントの作成（ユーザーの手動操作）を案内し、`gh auth login` の認証まで一緒に済ませてから先へ進む】

**Q3-3.** （チームで使う場合）「メンバーは何人ですか？また、メンバーによって見せる範囲を分けたいリポジトリはどれですか？」

【リポジトリごとのアクセス権方針を分割案の表に反映する。ワークスペースの同期スクリプトは「権限のないリポジトリを自動でスキップする」ため、メンバーごとに自分に見える範囲だけのワークスペースが作られる、という設計を説明する】

## フェーズ4：運用ルール

**Q4-1.** 「AIの応答言語は日本語でよいですか？」

**Q4-2.** 「業務の中で『何度も使う原理原則・ノウハウ』として貯めていきたい領域はありますか？（例の分類：営業、ライティング、分析、業務改善など、あなたの業務に合わせて）」

【回答に応じてナレッジ専用リポジトリ `repo_[会社/チーム名]-knowledge` と領域別ファイルの構成を決める。ナレッジリポジトリの本質は「チームで共有できるAIの記憶」（`tutorial/04-knowledge.md` 参照）。ひとり運用の場合は「当面はAIのメモリ機能でも代替できるが、チーム共有を始めるときに価値が立ち上がる」と正直に伝えた上で、作るかどうかを選んでもらう。不要と言われたら省略してよい】

**Q4-3.** 「Gitに入れたくない・入れられないファイルを手元に置く必要はありますか？」

【先に原則を伝える：ファイルは基本、該当する業務リポジトリの中に置く（文脈・履歴がつき、チームにも共有される）。ローカル専用フォルダ（プレフィックスなし）が必要なのは次の2ケースのみ——①大容量・一時ファイルなどGitに入れたくないもの ②それ自体が別のGitリポジトリであるものをまとめる場所（リポジトリの中にリポジトリは置けないため）。該当がなければ作らず、「必要になったら後から作ればよい」と伝える】
