## フェーズ2：業務領域の洗い出し（リポジトリ分割の設計）

このフェーズが設計の核心。時間をかけてよい。

**Q2-1.** 「あなた（またはあなたの会社）の業務を大きな塊で分けると、どんな領域がありますか？事業・部門・業務の種類など、思いつくまま挙げてください」

【回答を受けて深掘りする。目安：1つの領域に「扱う情報の機密度」「関わる人」「業務の文脈」の違うものが混ざっていれば分割を提案し、逆に細かすぎる場合（1領域に数ファイルしか入らない見込み）は統合を提案する】

**Q2-2.** 「クライアント（顧客）ごとに管理したい情報はありますか？」

【ある場合：`client-[英語名]` 命名規則を適用し、クライアントごとに独立リポジトリとする方針を説明する。クライアント情報は機密性が高く、担当者だけがアクセスできる形にすべきという理由も添える】

**Q2-3.** 「財務・人事・個人の記録など、特に機密度が高く、見られる人を限定したい領域はありますか？」

【ある場合：独立リポジトリに分離する。チーム利用ならアクセス権を個別設定できることを説明する】

【フェーズ2の締め：回答をもとにリポジトリ分割案（リポ名・ローカルフォルダ名・内容・アクセス範囲の表）を作成して提示し、合意を得る。原則は3つ — ①リポジトリ＝業務領域の境界 ②機密性の違うものは同居させない ③同一階層には同一粒度】

## フェーズ3：Git / チーム

**Q3-1.** 「GitHubの利用状況を教えてください」
1. 会社・チームの GitHub Organization がある
2. 個人アカウントのみ
3. GitHub は使わない（ローカルのみで管理）

【3の場合：各リポジトリは `git init` のみ（リモートなし）で構築する。マニフェスト（repos.tsv）と同期スクリプトは省略し、「チーム利用やバックアップが必要になったら後から導入できる」と伝える】

**Q3-2.** （Q3-1が1または2で、チームで使う場合）「メンバーは何人ですか？また、メンバーによって見せる範囲を分けたいリポジトリはどれですか？」

【リポジトリごとのアクセス権方針を分割案の表に反映する。ワークスペースの同期スクリプトは「権限のないリポジトリを自動でスキップする」ため、メンバーごとに自分に見える範囲だけのワークスペースが作られる、という設計を説明する】
